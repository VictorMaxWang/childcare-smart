# Visual Acceptance Criteria

- The final refactor is not a pixel-perfect recreation of every design image.
- The final refactor must feel like one unified SmartChildcare product.
- Existing business features, data flows, routes, role permissions, and demo account entry points must remain intact.
- Director, teacher, and parent experiences must stay meaningfully different:
  - Director: data, risk, management, weekly report, AI decision support.
  - Teacher: high-frequency daily records, class operations, consultation, voice/AI assistance.
  - Parent: child status, readable trends, feedback, storybook, mobile-first review.
- Pages must be real, usable UI, not static image mockups.
- Desktop, tablet, and mobile layouts must be reasonable.
- No obvious misalignment, overflow, overlap, clipped text, blocked controls, or newly introduced garbled text.
- Tables, forms, dialogs, empty states, and error states must share the same visual language.
- `artifacts/refactor-design-assets/` must remain a local reference source, not a production runtime dependency.
